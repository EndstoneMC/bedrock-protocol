# Tests

**Enforced by review.** `.claude/PROGRESS.md` (its Golden column) tracks which
packets still lack a golden.

## Every packet gets a golden round-trip test

Serialize the struct and assert its bytes against a
`const std::vector<std::uint8_t> golden`. Golden tests cover packets only -- a
non-packet component type is exercised through a packet that embeds it, not on its
own.

- Never hand-compute a golden, not even by reading the `Marshal` source. Construct the
  equivalent `packet.<Name>` in Sandertv/gophertunnel, marshal it through a
  `protocol.Writer` (this writes the body only, with no packet-id header, matching our
  `serialize`), and embed the resulting bytes. Precede the `golden` with two comment
  lines: a first line `// generated by gophertunnel:` and a second line with the
  marshalled `packet.<Name>{...}` literal. If a packet has no gophertunnel equivalent,
  raise it (see `protocol.md`, Stop and raise) -- do not hand-write the golden.
- A golden for a protocol version older than gophertunnel's current one cannot come
  from `master`: a packet's `Marshal` there reflects only the latest protocol. Check
  out gophertunnel at the historical commit whose marshaller matches that era (find it
  with `git log -- minecraft/protocol/packet/<name>.go`), marshal there, and record
  the commit: `// generated by gophertunnel @<commit>:`. Commits before the 2023
  Marshal/Unmarshal merge use a different API (`Marshal(buf *bytes.Buffer)`), so the
  driver differs per era.
- For a packet whose body is or contains a `CompoundTag`, the Go-side NBT struct used
  to build the golden must declare its `nbt:"..."`-tagged fields in alphabetical key
  order, and a `map[string]any` payload must be replaced by such a struct. Our
  `CompoundTag` is `std::map`-backed and writes entries in sorted-key order, while
  gophertunnel's NBT encoder follows Go struct declaration order and randomizes map
  iteration. Mismatch yields the same compound semantically but a different byte
  sequence, and the round-trip `REQUIRE(buf == golden)` fails. Sort the Go struct
  fields before generating.

## Tests are usage examples

The `tests/` folder doubles as the canonical place a new dev reads to learn the
library. Keep tests terse and high-signal, and demonstrate each public API shape.
Avoid:

- Enumerating many similar values (`REQUIRE(Foo::A == 0); REQUIRE(Foo::B == 1);
  ...`). Two or three anchors are enough: the first, a representative middle, and the
  latest.
- `struct Case { ... }` plus a vector plus a for-loop when the cases collapse to a
  single round-trip exercising the same path.
- Multi-line explanatory comments restating what the test does or quoting reference
  implementations. One short inline comment is enough.
- `SECTION` wrappers when each section is self-contained. Promote them to separate
  `TEST_CASE`s.
- Re-asserting state immediately after assigning it.
- A standalone "wire format" test for a primitive when an outer round-trip already
  exercises the same encoding.

Keep each public API shape demonstrated somewhere: explicit-version access
(`Foo_<N>`), bare-name-for-latest, field gating, serialize / deserialize, aggregate
init of generated structs, and optional-field handling. If consolidating drops the
only demonstration of a form, fold it into a different test.
